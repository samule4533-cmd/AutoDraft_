## 수행 작업

### 1. `src/rag_chain.py` — Phase 2 고도화

**chat_history 처리**
- `_trim_history()` 추가: 최근 N턴(기본 5턴) 슬라이딩 윈도우로 히스토리 제한
- `history_max_turns` RAG_CONFIG 항목 추가
- 선택 근거: 사내 문서 QA는 대화가 짧고 주제 전환이 잦아 최근 5턴으로 충분, 컨텍스트 창 초과 방지

**Query Reformulation**
- `should_reformulate()`: 지시어(`그럼`, `거기`, `그건` 등) 감지 + chat_history 존재 시만 True
- `reformulate_query()`: 이전 대화 맥락을 활용해 질문을 독립적으로 재작성
- 조건부 실행 이유: 모든 질문에 실행 시 매 턴 Gemini 호출 2회 → 비용/지연 2배
- 재작성 실패 시 원본 query 자동 fallback

**RagResult Pydantic 전환**
- `dataclass` → `Pydantic BaseModel`
- FastAPI `response_model` 선언 시 JSON 직렬화 자동화 + Swagger 자동 문서화
- `Citation`도 동일하게 전환

**신규 필드 추가**
- `used_query`: 실제 검색에 사용된 query (reformulation 결과 또는 원본)
- `reformulated_query`: 재작성이 발생한 경우만 값 존재 (디버깅용)

### 3. `src/rag_chain.py` — threshold 0.5 → 0.55 조정

**원인 분석**
- `【청구항 1】에 대해서 말해줘` 질문이 "사내 문서에 없습니다" 반환
- 실측: `【청구항 1】` 청크 distance = 0.5839, `# 【청구범위】` distance = 0.5039
- MiniLM 임베딩이 `【】` 괄호 표기를 의미적으로 잘 처리하지 못하는 문제

**threshold 비교 분석**
| threshold | 특허명 직접 질문 | 청구항 괄호 질문 | 서울날씨 (무관) |
|-----------|----------------|-----------------|-----------------|
| 0.50 | 통과 | 0개 통과 (차단) | 0개 (정상 차단) |
| 0.55 | 통과 | 5개 통과 (해결) | 0개 (정상 차단) |
| 0.60 | 통과 | 5개 통과 | 0개 (마진 축소) |

**결론**: 0.55가 괄호 표기 문제 해결과 노이즈 방지를 동시에 만족하는 적정값

### 4. `frontend/` — React 챗봇 프론트엔드 신규 구현

**기술 스택**: React 19 + TypeScript + Vite

**구현 기능**
- 사용자 질문 입력 → `POST /chat` API 호출 → 답변 표시
- `react-markdown` 적용: 볼드, 목록 등 마크다운 렌더링
- 로딩 애니메이션 (점 3개 깜빡임)
- 출처(citation) 태그 표시
- 채팅 기록 localStorage 저장/복원/삭제
- 사이드바 대화 목록 (토글 가능, 스크롤)
- 대화 목록 검색 + 날짜 표시 (오늘/어제/M.D)
- 채팅방 제목 더블클릭 인라인 편집
- 빈 상태: 입력창 화면 중앙 배치 → 첫 질문 후 하단으로 이동
- 헤더 중앙에 현재 채팅방 제목 표시 (bold)
- 응답 완료 후 입력창 자동 포커스

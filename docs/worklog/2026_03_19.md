## 수행 작업

### 1. `src/query_processor.py` — 쿼리 전처리 레이어 신규 분리

**분리 배경**
- `rag_chain.py`가 전처리와 실행을 모두 담당하면 비대해지고 디버깅이 어려움
- reformulation이 잘못됐는지, 검색이 잘못됐는지 원인을 분리해서 볼 수 있도록 분리

**구현 내용**
- `QueryContext` Pydantic 모델: 전처리 결과를 담는 단일 반환 타입
- `trim_history()`: 최근 N턴 슬라이싱 (기본 5턴)
- `should_reformulate()` / `reformulate_query()`: 지시어 감지 시 조건부 LLM 재작성
- `should_understand()` / `understand_query()`: 구어체/짧은 질문 키워드 변환
- `classify_query()`: meta / existence / content 패턴 기반 라우팅 (LLM 호출 없음)
- `process_query()`: 전처리 메인 진입점

**META_PATTERN 수정**
- `"파일 갯수만 알려줘"` 가 content로 오라우팅되는 버그 발견
- `갯수`, `개수` 패턴 추가로 수정

### 2. `src/rag_chain.py` — 버그 수정 및 개선

**chat_history 미전달 버그 수정**
- `_handle_content(ctx, req_id, filters)` 호출 시 `chat_history` 파라미터 누락
- `build_prompt()`에 `None`이 전달되어 대화 맥락이 LLM에 반영되지 않던 버그
- `_handle_content` 시그니처에 `chat_history` 추가, `ask()`에서 정상 전달하도록 수정

**rate limit 즉시 반환**
- 기존: rate limit 오류 시 60초×3회 sleep (사용자 3분 대기)
- 수정: 즉시 raise → llm_error fallback 반환

**meta 응답 청크 수 제거**
- "현재 8개 문서, 총 N개 청크" → "현재 8개 문서" (청크 수는 사용자에게 불필요한 정보)

**출처 형식 변경**
- 기존: 문장마다 `[출처: 헤더명]` 인라인 표기 → 읽기 불편
- 수정: 본문은 출처 없이 깔끔하게, 답변 하단에 `📎 출처` 섹션으로 모아서 표기
- 프롬프트 규칙 변경으로 처리

**existence threshold 분리**
- `"리트로핏 관련 문서 있나?"` → "등록되어 있지 않습니다" 오판 발견
- 원인: existence 쿼리가 content와 동일한 threshold(0.55) 사용
- `existence_distance_threshold: 0.65` 별도 추가 (존재 확인은 더 넓게 허용)

# llm_api.py

## 개요
`llm_api.py`는 Gemini API 호출을 담당하는 공통 유틸리티 모듈이다.  
프로젝트 내 다른 파일이 Gemini SDK를 직접 다루지 않도록 감싸는 래퍼 계층이며, 인증 처리, 클라이언트 생성, 이미지 입력 변환, JSON 응답 복구 및 예외 표준화를 담당한다.

이 파일의 핵심 목적은 다음과 같다.

- Gemini API 키 존재 여부 확인
- Gemini 클라이언트 싱글턴 생성 및 재사용
- PIL 이미지를 Gemini 입력 파트로 변환
- Gemini 응답을 안전하게 JSON으로 복구
- 인증 오류와 일반 API 오류를 구분하여 처리

즉, 이 모듈은 **Gemini 모델 호출을 프로젝트 전반에서 일관되게 사용할 수 있도록 추상화한 공통 인터페이스 계층**이다.

---

## 역할
이 파일은 Gemini API 사용에 필요한 공통 기능을 한 곳에 모아두는 역할을 한다.

구체적으로 다음 책임을 가진다.

- 환경 변수 기반 API 키 확인
- 인증 실패 상황을 명확한 예외로 분리
- Gemini 클라이언트의 지연 초기화(lazy initialization)
- 이미지 입력을 PNG 바이트 형태로 변환
- 응답 텍스트에서 JSON을 최대한 안전하게 복원
- 이미지 기반 JSON 생성 요청 수행

즉, 다른 모듈은 이 파일이 제공하는 함수만 호출하면 되고,  
Gemini SDK의 세부 호출 방식이나 예외 메시지 형식에 직접 의존하지 않아도 된다.

---

## 이 파일이 필요한 이유
직접 Gemini SDK를 여러 파일에서 반복 호출하면 다음과 같은 문제가 생길 수 있다.

- API 키 확인 로직이 중복된다
- 인증 실패와 일반 실패를 구분하기 어렵다
- JSON 응답 형식이 깨졌을 때 복구 로직이 중복된다
- 이미지 입력 포맷 변환 코드가 흩어진다
- 클라이언트 생성 방식이 파일마다 달라진다

`llm_api.py`는 이런 중복과 결합도를 줄이기 위해, **Gemini 호출과 관련된 공통 기능을 한 곳에 모아둔 인프라성 모듈**이다.

---

## 주요 구성 요소

### 예외 클래스

#### `GeminiAPIError`
Gemini 호출 전반에서 사용하는 기본 예외 클래스이다.  
Gemini 관련 실패를 프로젝트 내부에서 공통적으로 다루기 위한 상위 예외 역할을 한다.

#### `GeminiAuthError`
API 키 누락, 인증 실패, 권한 오류 등 인증 계열 문제를 나타내는 예외 클래스이다.  
호출 실패 중에서도 “인증 문제”를 별도로 식별하고 싶을 때 사용된다.

#### `GeminiResponseParseError`
Gemini 응답을 기대한 형식으로 해석하지 못했을 때를 위한 예외 클래스이다.  
현재 코드에서는 직접 raise되지는 않지만, 응답 파싱 오류를 명시적으로 분리하려는 의도가 반영된 구조이다.

---

## 전역 상태

### `_gemini_client`
Gemini 클라이언트를 한 번만 생성하고 재사용하기 위한 전역 변수이다.

#### 목적
- 매 호출마다 클라이언트를 새로 만들지 않도록 함
- 프로젝트 내부에서 동일한 클라이언트 인스턴스를 공유
- SDK 초기화 비용과 중복 생성을 줄임

이 구조는 사실상 **싱글턴 캐시**처럼 동작한다.

---

## 주요 함수

### `is_gemini_available() -> bool`
Gemini API 키가 현재 환경 변수에 설정되어 있는지 확인하는 함수이다.

#### 반환값
- `True`: `GEMINI_API_KEY`가 존재하고 비어 있지 않음
- `False`: API 키가 없음

#### 사용 목적
LLM 기능 사용 가능 여부를 사전에 점검하거나, 초기 진단 로직에서 활용할 수 있다.

---

### `get_api_key() -> str`
환경 변수에서 Gemini API 키를 읽어 반환하는 함수이다.

#### 동작 방식
- `GEMINI_API_KEY`를 읽음
- 값이 비어 있으면 `GeminiAuthError` 발생
- 정상 값이면 문자열 반환

#### 사용 목적
API 키 누락을 조기에 명확히 감지하고, 이후 호출 단계에서 अस्पष्ट한 오류가 발생하는 것을 줄이기 위함이다.

---

### `get_client() -> genai.Client`
Gemini 클라이언트를 반환하는 함수이다.

#### 동작 방식
- `_gemini_client`가 없으면 새 클라이언트 생성
- 이미 있으면 기존 인스턴스 재사용
- 결과적으로 클라이언트는 한 번만 초기화됨

#### 사용 목적
Gemini SDK 클라이언트 생성을 중앙화하고, 프로젝트 전반에서 일관된 방식으로 접근하게 하기 위함이다.

#### 특징
- lazy initialization 방식
- API 키 검증은 내부적으로 `get_api_key()`를 통해 수행됨

---

### `pil_to_png_part(pil_img: Image.Image) -> types.Part`
PIL 이미지를 Gemini 입력용 PNG 파트로 변환하는 함수이다.

#### 동작 방식
1. `BytesIO` 버퍼 생성
2. PIL 이미지를 PNG 포맷으로 저장
3. 바이트 데이터를 `types.Part`로 감싸 반환

#### 사용 목적
Gemini 멀티모달 입력 형식에 맞게 이미지를 안전하게 전달하기 위함이다.

#### 특징
- 입력은 `PIL.Image.Image`
- 출력은 Gemini SDK의 `types.Part`

---

### `safe_json_load(raw_text: str) -> Dict[str, Any]`
Gemini 응답 텍스트를 가능한 한 안전하게 JSON으로 변환하는 함수이다.

#### 목적
LLM 응답은 항상 완벽한 JSON 형식을 보장하지 않기 때문에,  
여러 복구 전략을 적용해 최대한 구조화된 결과를 얻기 위함이다.

#### 처리 순서

##### 1. 원문 그대로 JSON 파싱
응답 전체가 정상 JSON이면 바로 `json.loads()` 수행

##### 2. fenced code block 내부 JSON 추출
다음과 같은 형식을 우선 복구 시도

```json
{
  "key": "value"
}
```

예외 및 주의사항
1. API 키 없으면 호출 불가
GEMINI_API_KEY가 없으면 GeminiAuthError가 발생한다.
따라서 실행 전 .env 또는 환경 변수 설정이 필수다.

2. safe_json_load()는 복구 중심 구조
현재 구현은 JSON 파싱 실패 시 예외를 던지기보다 fallback dict를 반환한다.
즉, 호출부에서는 “파싱 실패 여부”를 summary나 반환 구조로 해석해야 한다.

3. GeminiResponseParseError는 현재 미사용
응답 파싱 오류를 위한 예외 클래스가 정의되어 있지만, 현재 로직에서는 직접 사용되지 않는다.
즉, 향후 구조화 응답 검증을 더 엄격하게 만들기 위한 확장 포인트로 볼 수 있다.

4. 이미지 전용 생성 함수 중심
현재 이 파일의 핵심 공개 함수는 generate_image_json()이며, 일반 텍스트 전용 생성 함수는 포함되어 있지 않다.
따라서 텍스트 기반 RAG 응답 생성은 다른 모듈에서 직접 Gemini를 호출하거나, 추후 이 파일에 텍스트 생성 래퍼를 추가할 수 있다.
